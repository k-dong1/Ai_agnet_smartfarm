# 공공데이터 사용 가이드

## 1. 프로젝트 기준

이 프로젝트는 **토마토 스마트팜 병충해 위험도 판단 AI Agent**를 만든다.
조교 요구사항에 따라 **더미데이터는 사용하지 않는다.**
모델 입력 데이터는 반드시 공공데이터 API 또는 공공데이터 파일에서 가져온 실제 데이터를 기반으로 구성한다.

최종 목표는 다음 흐름이다.

```text
공공데이터 API 호출
→ 응답 JSON/XML 저장
→ 필요한 필드 파싱
→ MLP 입력 feature 생성
→ 위험도 라벨 생성
→ PyTorch MLP 학습/예측
→ Agent가 원인 분석 + 행동 추천
→ Ollama 로컬 LLM이 설명문 생성
→ HTML 리포트 출력
```

## 2. 필수로 쓰는 데이터

### 필수 1: 스마트팜 환경/생육 데이터

#### 추천 데이터

**농림축산식품부_스마트팜 데이터 마트 품목별 데이터셋 제공 서비스**

- 공공데이터포털 URL: https://www.data.go.kr/data/15121325/openapi.do
- 용도: MLP의 실제 입력 데이터 생성
- 필요한 값:
  - 내부 온도
  - 내부 습도
  - CO2
  - 광량/일사량
  - 재배 작기 정보
  - 생육 단계 또는 조사 정보
- 이 데이터는 농가 기본정보, 작기정보, 온도·습도·CO2 등 환경정보, 양액·광·관수 등 제어정보, 생육정보를 포함한다고 설명되어 있으므로, 이 프로젝트의 핵심 데이터로 가장 적합하다.

#### 이 데이터를 쓰는 이유

MLP는 숫자 데이터를 입력받아 분류하는 구조다.
따라서 실제 환경 수치가 필요하다.
이 API는 스마트팜 환경값과 생육 관련 데이터를 제공하므로, 더미데이터 없이 MLP 입력 feature를 만들 수 있다.

따라서 병해충 API에서 제공하는 발생 환경과 방제 방법을 참고하여, 라벨링 규칙과 추천 행동 규칙을 만든다.

## 3. 선택적으로 쓰면 좋은 데이터

### 선택 1: 스마트팜 생산성 향상 모델 API

**농촌진흥청_스마트팜 생산성 향상 모델 오픈 API 조회 서비스**

- 공공데이터포털 URL: https://www.data.go.kr/data/15125691/openapi.do
- 용도: 토마토 생육 적정 조건 참고
- 제공 성격:
  - 작물별 생산성 향상 모델
  - 일사량별 모델
  - 생육상태별 모델
  - 시설유형별 모델
  - 생육시기별 모델

#### 활용 방식

이 API는 직접적인 센서 원천 데이터라기보다는, 토마토 생육 조건을 비교하는 기준 데이터로 쓰는 것이 좋다.

예시:

```text
현재 공공데이터 환경값
→ 토마토 생육 기준과 비교
→ 정상 범위 이탈 여부 판단
```

## 5. MLP 입력 feature 추천

공공데이터에서 바로 얻거나 계산해서 다음 7개 feature를 만든다.

```text
temperature        내부 또는 외부 기온
humidity           내부 또는 외부 습도
light              광량 또는 일사량
co2                CO2 농도, 없으면 결측 처리
humidity_duration  고습 상태가 연속된 시간 또는 연속 관측 횟수
temp_change        이전 시점 대비 온도 변화량
humidity_change    이전 시점 대비 습도 변화량
```

기존 기획에서 `pest_count`를 고려했지만, 공공데이터 API에서 실제 해충 개체 수가 제공되지 않을 가능성이 높다.
따라서 실제 API 데이터만 사용하는 조건에서는 `pest_count`를 무리하게 만들지 말고, 병해충 API는 개체 수가 아니라 **발생 조건/방제 정보 기준 데이터**로 사용한다.

최종 입력 feature는 다음 중 하나로 결정한다.

### 7-feature 버전

```text
temperature
humidity
light
co2
humidity_duration
temp_change
humidity_change
```

### 6-feature 버전

CO2가 없으면 다음처럼 구성한다.

```text
temperature
humidity
light
humidity_duration
temp_change
humidity_change
external_temperature
```

중요한 규칙:

```text
없는 값을 임의 랜덤값으로 만들지 않는다.
없는 값은 NaN 처리, 평균/중앙값 대체, 또는 feature 제외로 처리한다.
```

## 6. 라벨 생성 방식

공공데이터에는 보통 위험도 라벨이 없다.
따라서 실제 환경 데이터에 대해 규칙 기반으로 라벨을 생성한다.
이것은 더미데이터가 아니라 **실제 공공데이터에 위험도 기준을 적용하는 약지도 학습용 라벨링**이다.

라벨은 다음 4개를 사용한다.

```text
0 = 정상
1 = 주의
2 = 경고
3 = 심각
```

예시 규칙:

```text
습도 80% 이상: 위험 점수 +1
습도 90% 이상: 위험 점수 +1 추가
저온 조건: 위험 점수 +1
고습 지속 시간 증가: 위험 점수 +1
습도 급변: 위험 점수 +1
온도 급변: 위험 점수 +1
```

주의:

- 수치 임계값은 코드 상단 상수로 분리한다.
- 임계값의 근거는 병해충 검색 서비스의 발생 환경 설명과 프로젝트 발표 기준에 맞춰 주석으로 남긴다.
- 실제 생물학적 진단 시스템이 아니라 발표용 위험도 판단 프로토타입임을 README에 명시한다.

## 7. API 사용 시 주의사항

1. 공공데이터포털 회원가입 및 활용신청이 필요할 수 있다.
2. 서비스키는 코드에 직접 쓰지 않는다.
3. `.env` 또는 환경변수로 관리한다.
4. API 응답 원본은 반드시 `data/raw/`에 저장한다.
5. 파싱된 데이터는 `data/processed/`에 저장한다.
6. API 호출 실패 시 랜덤 데이터로 대체하지 않는다.
7. API 호출 실패 시 다음 중 하나를 수행한다.
   - 이전에 저장한 실제 API 응답 캐시 사용
   - 오류 메시지 출력 후 프로그램 종료
8. 캐시도 실제 API에서 받은 원본이어야 한다.
9. 임의 생성 데이터는 금지한다.

## 8. API 우선순위

Gemini CLI 또는 구현자는 다음 순서로 개발한다.

```text
1. 공공데이터 API 서비스키 설정
2. 스마트팜 데이터 마트 API 호출 성공 확인
3. raw JSON/XML 저장
4. 필요한 환경 필드 파싱
5. 병해충 검색 API 호출 및 원본 저장
6. 병해충 기준 정보 파싱
7. feature engineering
8. 라벨링 규칙 적용
9. MLP 학습
10. Agent 분석 및 HTML 리포트 생성
```
